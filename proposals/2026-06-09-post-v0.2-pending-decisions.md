## C. Open decisions for review (the "uncertain ones")

### C.1 — Runtime verification (#2) — THE ceiling
On natalie, 17 agents verified **compilation + plausibility**; the actual claims
(NO-DOUBLE-REGISTER, NO-DIALOG-STORM, bbox correctness) are **Rhino runtime behaviors no actor
or critic could execute** (no headless Rhino), all closed V2 `[model]` and deferred to "a human
in Rhino." Per first principle #2 ("the verifier is the whole game"), where the verifier is weak
the autonomy is theater — and no amount of correlated critics fixes it.

**Fork:**
- **(a) Build a headless-Rhino / recorded-macro test harness** so these claims become **V1
  machine-checkable** — turns the weakest verifier strong. High effort, highest payoff. (Caveat:
  your own stack notes RhinoCommon isn't thread-safe and GH is single-threaded STA — a harness is
  non-trivial.)
- **(b) Accept these as "draft + human-verifies-in-Rhino" missions** — and **stop paying** for
  3-lens panels + cold verifiers that provably can't reach the claim; redirect that budget into a
  sharper human-verification checklist.

Recommendation: decide (a) vs (b) before scaling more missions on Rhino code. Everything else is
deck chairs until this is set.

### C.2 — Is the per-node gate a gate or an annotator? (#4b)
5 critics, **0 blockers**, every real defect (unguarded `Add` sites; an unreachable #5 trigger)
ruled "criterion technically met → ACCEPT." Either (a) tighten the blocker bar so a genuine
acceptance-criterion violation *blocks* (the unguarded-Add finding arguably should have), and wire
the §3.3 gate-critic rebuttal; or (b) rename it honestly — it's a high-quality **annotator**, not
a gate. Pick one; the current in-between overstates what a green node means.

### C.3 — Parallelism (#6a)
Wiring `isolation:'worktree'` is a few lines, but it only pays if the **planner decomposes nodes
into disjoint file sets**. natalie's impl nodes overlapped on shared files, so worktrees would
just move the conflict to merge time. Decision: is node-level parallelism worth the planner
discipline + merge step for *coupled-subsystem* missions, or accept serial there and reserve
fan-out for genuinely independent work?

### C.4 — Seed the calibration corpus (#7)
Want me to write the first `classification_calibration` record from the natalie run (runtime-as-V2

`/mission-log-audit` on the natalie run and I transcribe the verdicts; (b) I draft a v0.1-tagged
